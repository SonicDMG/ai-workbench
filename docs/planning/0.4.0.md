# AI Workbench 0.4.0 — Release Plan

> **Status:** in progress on the long-lived `release/0.4.0` branch. This doc is the source of
> truth for the release — keep it current as slices land. Progress is tracked per-workstream below.

## Context

0.3.0 shipped HuggingFace retirement + chat/embeddings unified on the OpenAI-compatible protocol
(OpenRouter / OpenAI / Ollama). 0.4.0 is the next minor: **two headline features — agent
tool-calling and RBAC — on a hardening pass focused on security and reliability**, plus a broad
cleanup. Posture: *harden toward 1.0*. Breaking changes are allowed but each ships with
**Changed + Migration** notes (as 0.3.0 did). Large scope, no fixed timeline.

**Key finding — both headliners EXTEND mature foundations, they are not greenfield:**

- **Tool-calling already works.** `chat/agent-dispatch.ts` has a full sync + streaming tool-call
  loop (`MAX_TOOL_ITERATIONS = 6`), 7 built-in workspace tools (`chat/tools/registry.ts`), a shared
  dispatcher (`chat/tools/dispatcher.ts`), `tool-call`/`tool-result`/`token-reset` SSE events, and the
  OpenAI adapter plumbs `tools`/`tool_calls`. `AgentRecord.toolIds` exists but is **ignored** by
  `resolveAgentChat` (hardcodes `const tools = DEFAULT_AGENT_TOOLS`). Missing: per-agent allow-list,
  an MCP **client** (only the server façade exists), native tools, an Astra tool, per-tool
  audit/timeout, and UI cards.
- **Scopes already enforced.** `auth/authz.ts` ships `assertScope`/`requireScope`/
  `mutatingRouteWriteScope` gating `read`/`write` on API keys **today**. RLAC `PrincipalRecord` +
  `principal-resolver` thread identity onto `AuthSubject.principal`; `$principal.admin` is already a
  concept. Missing: **roles** (viewer/editor/admin), a **`manage`** tier for admin-only ops,
  OIDC-group→role mapping, and UI/CLI surfacing.
- **6 changesets already merged but unreleased** (MCP write tools, OIDC device flow, RLAC audit-shape
  stability, what's-new modal, 2 CLI fixes) — they ride in 0.4.0.

## Workstream A — Agent tool-calling (Headliner 1)

Extend the existing loop. Each item is independently shippable onto `release/0.4.0`.

### A1. Per-agent tool allow-list (foundation — do first)
- **Today:** `chat/agent-resolution.ts:126` hardcodes `const tools = DEFAULT_AGENT_TOOLS`;
  `AgentRecord.toolIds` (`control-plane/types.ts:528`) is stored but unused in dispatch.
- **Build:** make `resolveAgentChat` honor `toolIds`. Grandfather rule:
  - `toolIds` empty → all built-in workspace tools (current 7), no external/native.
  - `toolIds` non-empty → exactly that set, resolved across built-in + external-MCP + native registries.
  - External-MCP + native tools are never included unless explicitly listed (default-deny for high-risk).
- **Build:** a unified tool resolver (replaces the static `resolveTool`/`resolveDefaultTool`) composing
  built-in → workspace remote-MCP (A2) → native (A3) → Astra (A4), keyed by a namespaced id
  (`mcp:{serverId}:{tool}`, `native:fetch`, `astra:data_api`).
- **Files:** `chat/agent-resolution.ts`, `chat/tools/registry.ts`, `chat/tools/dispatcher.ts`; agent
  CRUD route + agent form UI for picking tools.
- **Verify:** `toolIds:[]` → 7 built-ins; `toolIds:["search_kb"]` → only that; external/native only when listed.

### A2. External MCP servers (client) + per-workspace registry
- **Today:** no MCP client; `@modelcontextprotocol/sdk@^1.29.0` in deps (server-side only).
- **Build:** `chat/tools/mcp-client.ts` (Streamable HTTP; `tools/list` → adapt to `AgentTool`; `tools/call`).
  Reuse `safeFetch`/`EndpointBaseUrlSchema` SSRF guard; resolve credential via `SecretResolver`.
- **Build:** `wb_config_mcp_tools_by_workspace` table + repo (memory/file/astra, mirror
  `repos/principals.ts`) + REST CRUD `/api/v1/workspaces/{w}/mcp-servers` + settings UI. Fields: id,
  label, url, credentialRef?, enabled, allowedTools?.
- **Verify:** CRUD conformance scenario; in-memory MCP-server round-trip (`InMemoryTransport`).

### A3. Native tools — HTTP fetch + web search (NO code-exec)
- **Build:** `chat/tools/native/fetch.ts` (`native:fetch`; `safeFetch`, timeout, size cap, content-type
  allow-list) + `web-search.ts` (`native:web_search`; pluggable provider behind a config key, off when
  unconfigured).
- **Defer:** code execution — documented non-goal for 0.4.0 (needs a real sandbox).
- **Files:** `chat/tools/native/*`, register in resolver (A1), `config/schema.ts` `chat.tools` block.
- **Verify:** SSRF rejection (private IP, redirect-to-internal), timeout, oversized body.

### A4. Astra Data API tool
- **Build:** `chat/tools/native/astra-data-api.ts` (`astra:data_api`) — scoped, read-mostly Data API
  query reusing the Playground/`search-dispatch.ts` path; emit `AstraQuerySnapshot` via the effects sink;
  astra/hcd only; refuse writes.
- **Verify:** unit test on a fake Astra `Db`; snapshot surfaces; writes refused.

### A5. Guardrails (ship with A1–A4)
- **Audit:** add `tool.invoke` to `AuditAction` (`lib/audit.ts`) + `docs/audit.md` row; emit from the
  dispatcher (args omitted, per the MCP `onToolInvoke` convention) with success/failure/denied.
- **Timeouts + size caps:** per-tool-call timeout (AbortController) in `dispatcher.ts`; output-size cap.
- **SSRF:** all outbound (A2, A3) via `safeFetch`. **Allow-list:** enforced by A1.

### A6. Chat UI — inline tool cards
- **Build:** extend the SSE consumer in `apps/web/src/pages/chat/` to render an expandable card per tool
  call (name, args summary, result, status) interleaved with tokens; respect `token-reset`; reuse the
  markdown render for result bodies.
- **Verify:** web vitest for the SSE→cards reducer; manual preview run.

## Workstream B — RBAC (Headliner 2)

Roles layered over the existing scope system. The authz negative-path suite (C1) is built alongside.

### B1. Role model + `manage` scope
- Roles → scopes: `viewer=[read]`, `editor=[read,write]`, `admin=[read,write,manage]`. New `manage` scope.
- **Split admin ops out of the blanket write gate.** `auth/authz.ts:mutatingRouteWriteScope` currently
  gates all mutations to `write` (so an editor key can issue API keys today). Carve out admin-only routes
  to require `manage`: **API keys CRUD, principals CRUD, RLAC policy + `rlacEnabled` toggle, workspace
  delete** (finalize the exact list during impl). KB/document/agent/service CRUD + ingest stay `write`.
  → **Migration note:** write-only keys lose access to these admin ops.
- **Files:** `auth/authz.ts`, affected route modules, `docs/auth.md`.

### B2. Unify identity (RLAC principal = RBAC subject)
- First-class `role` on `PrincipalRecord` (`control-plane/types.ts:447`, additive; 3 backends). Keep
  `attributes.admin` working for the RLAC DSL (derive from `role==='admin'`).
- Resolve role onto the subject in `auth/principal-resolver.ts`; effective scopes derive from role when
  the subject brought none; OIDC subjects map via B3.
- **Files:** `control-plane/types.ts`, `repos/principals.ts` (+3), `auth/principal-resolver.ts`,
  `auth/types.ts`, `routes/api-v1/principals.ts`.

### B3. OIDC groups/claims → role
- `auth.oidc.roleMapping` config in `config/schema.ts`; apply in `auth/oidc/claims.ts:subjectFromClaims`;
  default `viewer` when no mapping/claim. **Files:** `config/schema.ts`, `auth/oidc/claims.ts`, `docs/auth.md`.

### B4. Scoped/role-bearing API keys + UI ✅
- Keys already carry `scopes` (default `['read','write']`). Issue by role (derive scopes) or explicit
  scopes; persist both; add a scope/role picker to the API-keys panel.
- **Files:** `routes/api-v1/api-keys.ts`, `auth/apiKey/*` (no token-format change), web API-keys UI.
- **Shipped (web):** reworked the API-key issue dialog (`CreateApiKeyDialog`) from a 2-preset scope
  toggle to a **3-role picker** (Viewer → `[read]`, Editor → `[read,write]`, Admin →
  `[read,write,manage]`), defaulting to Editor (preserves the pre-roles default). `ApiKeysPanel`
  renders each key's scopes as a single role chip (Viewer/Editor/Admin, Admin in red) and falls back
  to per-scope chips for unrecognized sets. Added `manage` to the web `ApiKeyScopeSchema` + a drift
  test against the generated OpenAPI enum; regenerated `api-types.generated.ts` (now carries `manage`
  + the `Role` enum on principals). Backend route/schema were already done in B1–B3 — no runtime edits.

### B5. Enforce at MCP, CLI, UI ✅
- **MCP:** `mcp/server.ts` already gates writes via `denyIfMissingScope(...,"write",...)`; extend
  admin-shaped tools to `manage`. **CLI:** translate 403/scope errors into role guidance. **UI:** expose
  role from `/auth/me`; `useRole()` hook; hide/disable admin actions for non-admins.
- **Files:** `mcp/server.ts`, `packages/aiw-cli/src/*`, `routes/auth.ts`, `apps/web/src/hooks/`, action sites.
- **Shipped:**
  - **/auth/me** now returns the caller's effective `role` + `scopes` (both `null` = unscoped). Pure
    projection; HTTP routes stay authoritative. (`routes/auth.ts` + `flow.test.ts`.) **No conformance
    fixture regen needed** — `/auth/me` isn't a conformance scenario (drift guard stays green).
  - **UI:** `useRole()` hook reads `/auth/me`; `WorkspaceSettingsPage` hides API-key management, the
    RLAC toggle, principal + policy panels, and the workspace-delete button for non-admins
    (permissive default when no role signal). B4 owns the issue-dialog internals; B5 only gates the
    visibility of admin entries.
  - **CLI:** `aiw whoami` / `aiw login` surface role + scopes; the central error handler translates a
    `403 forbidden` "missing required scope" into role guidance (`hintForForbidden`), mirroring the
    401 login guidance.
  - **MCP:** consistency check only — the exposed tools are reads + content writes (gated on `write`);
    there are **no admin-shaped MCP tools** (no api-key / principal / policy / workspace-delete tool),
    so nothing graduates to `manage`. No change required.

## Workstream D — Reliability hardening

### D1. SQLite control-plane driver
- **Today:** `control-plane/file/state.ts` rewrites whole JSON per mutation; `file/chat-messages.ts`
  appends by full-file rewrite (quadratic under streaming load).
- **Build:** `control-plane/sqlite/` implementing `ControlPlaneStore` (mirror `file/*` slices); register
  `driver:"sqlite"` in `factory.ts` + `config/schema.ts` (`{ driver:"sqlite", path }`). Row-level WAL.
- **Lib:** recommend `better-sqlite3` (stable, sync); evaluate Node 22 `node:sqlite` (no native build →
  simpler Docker image). Decide in the impl PR; note multi-stage Docker impact.
- **Verify:** the SQLite store must pass the same control-plane + jobs + chat-messages contract suites.

### D2. Job durability for all kinds
- **Today:** only async-ingest snapshots + resumes; sweeper marks other orphans failed.
- **Build:** `JobKind`→resume-callback registry; each kind persists its input snapshot; sweeper replays
  idempotently. **Files:** `jobs/types.ts`, `jobs/sweeper.ts`, `jobs/store.ts` (+3), `tests/jobs/contract.ts`.

### D3. Streaming robustness
- Shared SSE helper (`lib/sse.ts`); `Last-Event-ID`/resume where feasible; ensure cancel aborts the
  in-flight LLM fetch; harden partial-write recovery (`emitTerminalError`). Refactor jobs + agents routes.

### D4. Chat-surface conformance
- **Today:** every surface has fixtures except chat; `FixtureChatService` (`chat/fixture.ts`) exists.
- **Build:** `chat-message-crud-basic` + `chat-sse-stream` scenarios (drive `FixtureChatService` via a
  `chatScript`); regenerate fixtures; cover the SSE happy path (`user-message`→`token`*→`done`).

## Delivery & sequencing

**Branch:** long-lived `release/0.4.0` off `main`. Slices land as PRs into `release/0.4.0`; rebase on
`main` periodically; merge the whole release to `main` at the end and tag. Dependency upgrades (E2) and
the queued changesets (F) may also land on `main` directly and be merged down.

**Build order:**
1. **B1–B2 RBAC core** (roles + `manage` + unified identity) — foundational; tool guardrails + MCP gating
   depend on it. Build **C1** in lockstep.
2. **B3–B5** OIDC mapping, scoped-key UI, MCP/CLI/UI enforcement.
3. **A1 allow-list** → **A5 guardrails** → **A2/A3/A4** (parallel once A1 lands) → **A6 UI cards**.
4. **D1–D4** independent; run in parallel throughout.
5. **E** continuous; **C2** continuous.
6. **F** fold changesets, migration notes, version bump, merge → `main`, tag `v0.4.0`.
